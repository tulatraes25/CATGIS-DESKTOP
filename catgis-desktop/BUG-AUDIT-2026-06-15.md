## CRITICAL

### BUG-001: Resource Leak — FlatGeobufLoader FileInputStream Not Closed
- **File:** `FlatGeobufLoader.java`
- **Line:** ~219
- **Severity:** CRITICAL
- **Category:** Resource Leaks
- **Code:**
```java
try (FileChannel dataChannel = new FileInputStream(file).getChannel()) {
    dataChannel.position(dataOffset);
    ByteBuffer dataBuf = ByteBuffer.allocateDirect(capacity);
    dataChannel.read(dataBuf);
```
- **Problem:** `new FileInputStream(file)` is created inline and never assigned to a variable. When `getChannel()` is called, the `FileInputStream` reference is lost. If `getChannel()` succeeds but an exception occurs later in the try block, the `FileInputStream` is never closed. The try-with-resources only closes the `FileChannel`, not the underlying stream. This leaks a file handle on every FlatGeobuf load.
- **Fix:** Assign the `FileInputStream` to a variable and close it explicitly:
```java
try (FileInputStream dataFis = new FileInputStream(file);
     FileChannel dataChannel = dataFis.getChannel()) {
```

### BUG-002: Connection Leak — WfsCapabilitiesService Missing disconnect()
- **File:** `WfsCapabilitiesService.java`
- **Line:** ~31–48
- **Severity:** CRITICAL
- **Category:** Resource Leaks
- **Code:**
```java
HttpURLConnection connection = (HttpURLConnection) new URL(url).openConnection();
connection.setRequestMethod("GET");
// ...
int status = connection.getResponseCode();
if (status < 200 || status >= 300) {
    throw new IllegalArgumentException("El servicio respondio con codigo HTTP " + status + ".");
}
try (InputStream input = connection.getInputStream()) {
    // ... parse XML ...
}
```
- **Problem:** `connection.disconnect()` is never called. If the HTTP status check throws (non-2xx), the connection is leaked. Even on success path, the underlying socket is not released back to the connection pool. Same issue exists in `WmsCapabilitiesService.java` (identical pattern).
- **Fix:** Wrap in try-finally:
```java
HttpURLConnection connection = (HttpURLConnection) new URL(url).openConnection();
try {
    // ... existing code ...
} finally {
    connection.disconnect();
}
```

### BUG-005: Process Leak — PostgisConnectionStore getWindowsMachineGuid()
- **File:** `PostgisConnectionStore.java`
- **Line:** ~176–180
- **Severity:** HIGH
- **Category:** Resource Leaks
- **Code:**
```java
Process p = new ProcessBuilder(cmd).redirectErrorStream(true).start();
if (p.waitFor(3, TimeUnit.SECONDS) && p.exitValue() == 0) {
    String output = new String(p.getInputStream().readAllBytes()).trim();
    if (!output.isBlank()) return output;
}
```
- **Problem:** The `Process` is never destroyed. If `waitFor()` times out (returns false), the process is left running as a zombie. Even on success, `p.destroy()` is never called. This leaks OS processes, especially problematic since this is called during password encryption/decryption operations.
- **Fix:**
```java
Process p = new ProcessBuilder(cmd).redirectErrorStream(true).start();
try {
    if (p.waitFor(3, TimeUnit.SECONDS) && p.exitValue() == 0) {
        String output = new String(p.getInputStream().readAllBytes()).trim();
        if (!output.isBlank()) return output;
    } else {
        p.destroyForcibly();
    }
} finally {
    p.destroyForcibly();
}
```

### BUG-006: Thread Safety — MapPanel Mutable Fields Accessed from Multiple Threads
- **File:** `MapPanel.java`
- **Line:** ~85–135 (field declarations)
- **Severity:** HIGH
- **Category:** Thread Safety Issues
- **Code:**
```java
final Map<Layer, ShapefileData> shapefileLayers = new LinkedHashMap<>();
final Map<Layer, LocalRasterData> rasterLayers = new LinkedHashMap<>();
final Map<Layer, OnlineRasterSource> onlineTileLayers = new LinkedHashMap<>();
final Map<Layer, OnlineWmsLayer> onlineWmsLayers = new LinkedHashMap<>();
// ...
double viewMinX = 0;
double viewMinY = 0;
double zoomFactor = 1.0;
boolean dragging = false;
String currentTool = "MOVE";
Layer selectedLayer;
SimpleFeature selectedFeature;
final Map<Layer, List<String>> tableSelectionIds = new LinkedHashMap<>();
```
- **Problem:** All these fields use non-thread-safe `LinkedHashMap` and plain (non-volatile) primitives. The maps are read from background threads (tile fetchers, WMS cache, online layer renderers) and written from the EDT. The `selectedLayer`, `selectedFeature`, `viewMinX/Y`, `zoomFactor` fields are read/written from both EDT and background threads without synchronization. This can cause `ConcurrentModificationException`, stale reads, or rendering artifacts.
- **Fix:** Use `ConcurrentHashMap` for the layer maps, or synchronize access. Mark `viewMinX`, `viewMinY`, `zoomFactor`, `selectedLayer`, `selectedFeature` as `volatile`.

### BUG-007: Thread Safety — CatgisSocketServer Static Mutable State Without Synchronization
- **File:** `CatgisSocketServer.java`
- **Line:** ~33–37
- **Severity:** HIGH
- **Category:** Thread Safety Issues
- **Code:**
```java
private static ServerSocket serverSocket;
private static Thread listenerThread;
private static boolean running = false;
```
- **Problem:** `running` is not `volatile`. The listener thread reads `running` in a loop, but it's set from the main thread via `stop()`. Without `volatile`, the JVM may cache the value and the listener thread may never see the change, causing the server to not shut down cleanly.
- **Fix:** Mark `running` as `volatile`:
```java
private static volatile boolean running = false;
```

### BUG-008: Thread Safety — CatmapSocketClient Static State Without Synchronization
- **File:** `CatmapSocketClient.java`
- **Line:** ~20–24
- **Severity:** HIGH
- **Category:** Thread Safety Issues
- **Code:**
```java
private static Socket socket;
private static PrintWriter out;
private static BufferedReader in;
private static boolean connected = false;
```
- **Problem:** All fields are static and mutable with no synchronization. `connect()`, `disconnect()`, and `sendCommand()` can be called from different threads. `connected` is not volatile, so changes may not be visible across threads. Race conditions can cause `NullPointerException` when `sendCommand()` runs while `disconnect()` nulls out `socket`/`out`/`in`.
- **Fix:** Mark `connected` as `volatile`, and synchronize `connect()`, `disconnect()`, and `sendCommand()` on a shared lock object.

### BUG-009: NullPointerException — PostgisConnectionFactory Race Condition on Pool Creation
- **File:** `PostgisConnectionFactory.java`
- **Line:** ~43–47
- **Severity:** HIGH
- **Category:** NullPointerException Risks
- **Code:**
```java
HikariDataSource ds = poolCache.get(fingerprint);
if (ds == null || ds.isClosed()) {
    ds = createPool(info);
    poolCache.put(fingerprint, ds);
}
```
- **Problem:** This is a check-then-act race condition. Two threads with the same fingerprint can both see `ds == null`, both create a new pool, and one pool leaks (never closed). The `ConcurrentHashMap` only protects individual operations, not the compound check-then-act sequence.
- **Fix:** Use `computeIfAbsent` or synchronize:
```java
HikariDataSource ds = poolCache.computeIfAbsent(fingerprint, k -> createPool(info));
if (ds.isClosed()) {
    ds = createPool(info);
    poolCache.put(fingerprint, ds);
}
```

### BUG-011: Broad Exception Catch — BatchProcessorDialog Wraps All Exceptions as RuntimeException
- **File:** `BatchProcessorDialog.java`
- **Line:** ~200–203
- **Severity:** HIGH
- **Category:** Exception Handling Anti-patterns
- **Code:**
```java
default -> throw new RuntimeException("Operacion no soportada: " + operation);
} catch (Exception e) {
    throw new RuntimeException(e);
}
```
- **Problem:** The outer catch wraps ALL exceptions (including checked exceptions like `IOException`) as unchecked `RuntimeException` without adding context. This loses the original exception type and makes error handling upstream impossible. The `SwingWorker.done()` method then catches this generic RuntimeException.
- **Fix:** Let specific exceptions propagate, or wrap with meaningful context:
```java
} catch (IOException e) {
    throw new RuntimeException("Error procesando lote: " + e.getMessage(), e);
}
```

## MEDIUM

### BUG-012: Broad Exception Catch Pattern — 700+ catch(Exception) Blocks
- **File:** Multiple files (707 occurrences across codebase)
- **Line:** Various
- **Severity:** MEDIUM
- **Category:** Exception Handling Anti-patterns
- **Code:** Pattern appears in virtually every file, e.g.:
```java
} catch (Exception ex) {
    JOptionPane.showMessageDialog(this, "Error: " + ex.getMessage(), "Error", JOptionPane.ERROR_MESSAGE);
}
```
- **Problem:** Catching `Exception` instead of specific exception types swallows programming errors (NPE, ClassCastException, etc.) and presents them as user-facing "errors" without distinguishing recoverable from unrecoverable failures. The logging pattern `CatgisLogger.warn("X: operation failed", ignored)` provides no useful context about what actually failed.
- **Fix:** Catch specific exception types. For I/O operations, catch `IOException`. For parsing, catch `NumberFormatException`. Reserve `catch (Exception)` for top-level error boundaries only.

### BUG-013: Resource Leak — PostgisConnectionStore obfuscate() Silently Returns Empty on Crypto Failure
- **File:** `PostgisConnectionStore.java`
- **Line:** ~148–155
- **Severity:** MEDIUM
- **Category:** Exception Handling Anti-patterns
- **Code:**
```java
private static String obfuscate(String value) {
    try {
        // ... AES-GCM encryption ...
        return java.util.Base64.getEncoder().encodeToString(combined);
    } catch (Exception e) {
        return "";
    }

```
- **Problem:** If encryption fails (e.g., JCE not available, key derivation fails), the method returns an empty string. The caller then stores an empty string as the "encrypted" password.
```java
} catch (Exception e) {
    CatgisLogger.warn("PostgisConnectionStore: password encryption failed", e);
    return "";
}
```

### BUG-016: Resource Leak — CatmapSocketClient Static Socket Never Cleaned Up on Error
- **File:** `CatmapSocketClient.java`
- **Line:** ~32–40
- **Severity:** MEDIUM
- **Category:** Resource Leaks
- **Code:**
```java
public static boolean connect() {
    try {
        socket = new Socket("127.0.0.1", 8899);
        socket.setSoTimeout(5000);
        out = new PrintWriter(new OutputStreamWriter(socket.getOutputStream(), StandardCharsets.UTF_8), true);
        in = new BufferedReader(new InputStreamReader(socket.getInputStream(), StandardCharsets.UTF_8));
        String response = sendCommand("PING");
        connected = response != null && response.contains("\"ok\"");
        return connected;
    } catch (Exception e) {
        connected = false;
        return false;
    }
}
```
- **Problem:** If `new Socket()` succeeds but `getOutputStream()` or `sendCommand("PING")` fails, the socket is never closed. The catch block sets `connected = false` but doesn't close the socket or the streams. The static `socket` reference holds a leaked connection.
- **Fix:** Close socket in catch block:
```java
} catch (Exception e) {
    connected = false;
    disconnect(); // Close socket, out, in
    return false;
}
```

### BUG-017: Deprecated API — PBEKeySpec Not Cleared After Use
- **File:** `PostgisConnectionStore.java`
- **Line:** ~163–166
- **Severity:** MEDIUM
- **Category:** Deprecated/Unsafe API Usage
- **Code:**
```java
PBEKeySpec spec = new PBEKeySpec(
        passphrase.toCharArray(), salt, 100_000, 256);
return factory.generateSecret(spec).getEncoded();
```
- **Problem:** `PBEKeySpec` holds the passphrase in memory and is never cleared. The `clearPassword()` method should be called after `generateSecret()` to zero out the sensitive data. This leaves the passphrase in heap memory, accessible via memory dumps.
- **Fix:**
```java
PBEKeySpec spec = new PBEKeySpec(
        passphrase.toCharArray(), salt, 100_000, 256);
try {
    return factory.generateSecret(spec).getEncoded();
} finally {
    spec.clearPassword();
}
```

### BUG-018: Logic Bug — CRSDefinitions catch(Throwable) Hides Fatal Errors
- **File:** `CRSDefinitions.java`
- **Line:** ~212, ~215, ~400
- **Severity:** MEDIUM
- **Category:** Exception Handling Anti-patterns
- **Code:**
```java
} catch (Throwable t) {
    // ...
} catch (Throwable t2) {
```
- **Problem:** Catching `Throwable` catches `Error` subclasses like `OutOfMemoryError`, `StackOverflowError`, and `LinkageError` which should never be caught. This can mask fatal JVM errors and leave the application in an undefined state.
- **Fix:** Catch `Exception` instead of `Throwable`. If specific `Error` handling is needed (e.g., `NoClassDefFoundError` for optional dependencies), catch those specifically.

## Recommendations

1. **Immediate (CRITICAL):** Fix BUG-001, BUG-002, BUG-003 — resource leaks in FlatGeobufLoader and HTTP capability services. These cause file handle exhaustion and connection pool exhaustion under load.

2. **Short-term (HIGH):** Fix BUG-005 (process leak), BUG-006 (MapPanel thread safety), BUG-010 (copy-paste bug in layout composer). The thread safety issues in MapPanel are the most architecturally significant — consider using `ConcurrentHashMap` for layer maps and `volatile` for shared primitives.

3. **Medium-term:** Address the broad exception catching pattern (700+ instances) by establishing a convention for exception handling. Consider an `ExceptionUtils` helper that provides consistent logging with context.

4. **Long-term:** Audit all `ProcessBuilder` usages for proper process lifecycle management. Consider a shared `ProcessRunner` utility that handles timeout, stream consumption, and cleanup consistently.
